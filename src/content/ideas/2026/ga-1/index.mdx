## Divide & Conquer

Use the visualizer below to see how a recurrence shifts between the three Master Theorem cases and which level dominates the work.

### A 4-Step Writeup Template

Algorithm explanations are often clearer when they are separated into four parts instead of being collapsed immediately into code. Erickson's "What, How, Why, How fast" split is a good default. _The important point is that correctness does not come for free just because the steps sound plausible._

- **1. What: Problem Specification:** Explicitly define the problem the algorithm solves. Include the type and meaning of inputs, exactly what the output should be, and state any hidden assumptions.
- **2. How: Algorithm Description:** Explain _how_ the algorithm works using a plain-English narrative or a high-level structured outline.
  - Avoid writing code disguised as words (e.g., translating a script line-by-line). Deeply nested bullet points are usually a red flag.
  - Walk through every major step, ending with the final return statement.
  - If you are adapting a known algorithm, explicitly highlight the modifications you made.
- **3. Why: Justification of Correctness:** Explain _why_ your algorithm works.
  - Write a narrative, informal justification (no formal mathematical or inductive proofs are required).
  - If you modified a known algorithm, you must explain why your specific tweaks are correct and successfully solve the problem.
- **4. How Fast: Runtime Analysis:** Analyze the algorithm using **Big-O notation**, with worst-case analysis as the default unless some other model is stated explicitly.
  - Small $O(1)$ steps usually do not need to be analyzed individually.
  - You may assert the standard runtime of unmodified "black-box" algorithms without extra proof.
  - If you modify a black box, you must justify how those changes impact the runtime, even if the overall Big-O doesn't change.
  - Always conclude with a single, fully simplified final runtime in Big-O notation.

### Guide to finding a D&C solution

A practical workflow for drafting a divide-and-conquer solution is:

1. **Identify the Pattern:** Determine if your problem maps to a standard algorithm.
   - _Sorted data_ usually points to Binary Search.
   - _Unsorted data_ usually points to Merge Sort or FastSelect.
   - _Polynomials, convolution, or multiplication_ usually point to Fast Fourier Transform (FFT). For integer multiplication specifically, Karatsuba's algorithm is a classic approach ($O(n^{\log_2 3})$).
2. **Define Modifications (If Any):** Decide if you can use the known algorithm "as is" or if it needs tweaking. If you tweak it, clearly outline what steps, inputs, or outputs are changing.
3. **Draft the Steps:** Write out the algorithm's execution in clear words. Define the base cases if they are needed, and make sure the algorithm ultimately returns exactly what the prompt asks for.
4. **Draft the Proof of Correctness:** Build an informal justification. Helpful questions include:
   - _Why is this specific black box suited for this input data?_
   - _Why did you make those specific modifications?_
   - _Why are we guaranteed that the problem space shrinks on each round?_
   - _Why are the base cases logically sound?_
   - _What is the logic behind branching left or right on each recursive call?_
5. **Calculate the Runtime:** Step through the algorithm and summarize the final asymptotic running time.

### Master Theorem

For some constants $a > 0$, $b > 1$, and $d \geq 0$,
where:
$a$ - number of subproblems
$b$ - is the factor by which the input size shrinks
$d$ - is the exponent of the work done at each level of recursion (so combining costs $O(n^d)$)

$$
T(n) = \underbrace{aT (n/b)}_{\text{recursive branching}} + \underbrace{O(n^d)}_{\text{local work (root)}}
$$

$$
T(n) =
\begin{cases}
\Theta(n^{\log_b(a)}) &\text{if } n^{\log_b(a)} > n^d \quad\text{ ``leaf heavy"} \\
\Theta(n^d \log(n)) &\text{if } n^{\log_b(a)} = n^d \quad\text{ ``balanced"}\\
\Theta(n^d) &\text{if } n^{\log_b(a)} < n^d \quad\text{ ``root heavy"} \\
\end{cases}
$$

- **Case 1 (Leaf-Heavy):** $n^{\log_b(a)} > n^d$. The recursive branching explodes, meaning the vast majority of the work happens at the bottom of the tree. The leaves win, so the bound is strictly the number of leaves: $\Theta(n^{\log_b(a)})$.

- **Case 2 (Perfectly Balanced):** $n^{\log_b(a)} = n^d$. It is a tie. The local work at the root matches the work at the leaves, and the work across every internal level of the tree is identical. Because the work per level is constant, you just multiply the root work by the number of levels ($\log n$). The bound is: $\Theta(n^d\log n)$. _(Note: The "extended" Case 2 handles $f(n) = \Theta(n^{\log_b a} \log^k n)$, resulting in $\Theta(n^{\log_b a} \log^{k+1} n)$)._
- **Case 3 (Root-Heavy):** $n^{\log_b(a)} < n^d$. The local work dwarfs the branching. The work shrinks geometrically as you go down the tree, meaning the total time is dominated by the massive chunk of work done at the very beginning. The root wins, so the bound is strictly the root work: $\Theta(n^d)$.

### Work per level

1. **Number of nodes:** At level $k$, the tree has branched $k$ times, so there are $a^k$ nodes.
2. **Size of each subproblem:** The problem size $n$ has been divided by $b$ exactly $k$ times, making the input size for each node $n/b^k$.
3. **Work per node:** Since the local work is bounded by $O(\text{size}^d)$, the work at a single node on level $k$ is $(\frac{n}{b^k})^d$, which simplifies to $\frac{n^d}{b^{kd}}$.

If you multiply the **number of nodes** by the **work per node**, you get the total work for the entire $k$-th level:

$$
a^k \times \frac{n^d}{b^{kd}}
$$

Pull out the $n^d$ (which is the work at the root), and group the $k$ exponents together, and you get the formula for the total work done at the $k$-th level of the tree:

$$
O(n^d) \times \left(\frac{a}{b^d}\right)^k
$$

The Master Theorem uses $\log_b(a)$ instead of comparing $a$ directly to $b^d$, but the comparison is the same after you rewrite it.

If you take $a=b^d$ and take the $\log_b$ of both sides, you get:

$$
\log_b(a)=\log_b(b^d)
$$

$$
\log_b(a)=d
$$

So, comparing $\log_b(a)$ to $d$ is mathematically identical to comparing the branching factor $a$ to the work-shrinkage factor $b^d$.

### Example Walkthrough: Majority Element

Here is a concrete example in the 4-step template.

**1. What: Problem Specification**
An array `A[1…n]` has a majority element if more than half its entries are the same. Given an array, design an $O(n \log n)$ algorithm to determine if it has a majority element, and if so, find it. The elements are not sortable (e.g., GIF files), so you can only test for equality `A[i] == A[j]` in $O(1)$ time. Expected output is the majority element or `NO_MAJORITY` if none exists.

**2. How: Algorithm Description**
1. Recursively divide the input array into two equal-sized subarrays, $A_1$ and $A_2$.
2. **Base case:** If the subarray has size 1, report its single element as the majority element.
3. At each level of recursion, check the majority element returned by the left and right subarrays.
4. For each potential majority element, scan the combined array to count its occurrences.
5. If one element occurs strictly more than $n/2$ times in the combined array, return it as the majority element.
6. Otherwise, return `NO_MAJORITY`.

**3. Why: Justification of Correctness**
If a majority element exists in the combined array, it *must* be the majority element in at least one half. The recursion therefore surfaces the right candidate from below, and the counting step filters out any candidate that is not actually a majority for the current subarray.

**4. How Fast: Runtime Analysis**
- We split the problem into 2 subproblems: $a = 2$.
- Each subproblem is exactly half the size: $b = 2$.
- The work to combine results (scanning the array to count occurrences) takes linear time: $O(n^1)$, so $d = 1$.
- We have the recurrence $T(n) = 2T(n/2) + O(n)$.
- Since $\log_b(a) = \log_2(2) = 1$, and $d = 1$, we are in **Case 2 (Balanced)**.
- The overall runtime is $O(n^1 \log n) \implies O(n \log n)$.

### What to remember

- **Problem Structure**: State the algorithm, explain why it works, and end with one simplified Big-O bound.
- **Master Theorem Formula**: $T(n) = aT(n/b) + O(n^d)$
- **Master Theorem Cases**:
  - $d > \log_b(a)$: Root-heavy $\to O(n^d)$ (combine step dominates).
  - $d = \log_b(a)$: Balanced $\to O(n^d \log n)$ (work is spread evenly).
  - $d < \log_b(a)$: Leaf-heavy $\to O(n^{\log_b a})$ (recursive calls dominate).

Divide & Conquer splits a problem into independent branches. Backtracking uses recursion for a different reason: each branch is a different **choice**.

## Backtracking

In *Algorithms*, Jeff Erickson presents backtracking as a way to build a solution one decision at a time. When there are several legal next moves, the algorithm tries them and keeps the best outcome.

### Building Solutions Incrementally

Backtracking is not the main focus of this post, but it provides a cleaner lead-in to Dynamic Programming than jumping straight to tables and recurrences.

Divide & Conquer is about decomposition and combination: split the problem, solve the pieces, combine the results. Backtracking is about **search**. You keep a **partial solution**, ask what the next decision is, and recurse on each legal option.

Erickson's examples make it clear that this is not just about `n`-queens:

- In **`n`-queens**, the output is a sequence of queen positions, one per row.
- In a **game tree**, the output is a sequence of moves, one choice per game state.
- In **subset sum**, the output can be viewed as a sequence of include/exclude decisions over the input array.

The section is included mainly for that transition. Many DP problems make more sense once the brute-force recursive search hiding underneath them has been made explicit.

### The basic picture

Erickson opens with `n`-queens. Place queens one row at a time. On row $r$, try each column from left to right. If a square is attacked by an earlier queen, skip it. If it is legal, place a queen there and recurse on the remaining rows.

That example makes two things clear:

- Every recursive call is a **legal partial solution**.
- The full algorithm is just a **depth-first search of the recursion tree** of those partial solutions.

The design point Erickson adds is more useful: a recursive call needs not only the unprocessed part of the input, but also a **summary of earlier decisions**. The real question is how small that summary can be.

- For **`n`-queens**, you unfortunately need the previously placed queens in full detail.
- For a **game tree**, you often only need the current game state; the exact move history may not matter.
- For **subset sum**, you do not need the exact chosen subset, only the remaining items and the remaining target value.

Backtracking is not just "try stuff recursively." It is a search over an implicit tree where each node stores only the information needed to make the **next** decision. Pruning just means refusing to recurse into branches that are already invalid.

That also explains why exponential runtime is the default. Even with good pruning, the tree can still be huge.

It also explains when Backtracking does **not** naturally turn into DP. In `n`-queens, the full placement of earlier queens is so specific that two branches rarely reach the same state, so memoization buys little. In problems like subset sum, many different paths can lead to the same reduced state, such as "same remaining items, same remaining target." That overlap is what DP uses.

### Discovering repeated work

The useful step here is the **reformulation**. Once you ask for the smallest summary of past decisions, you often realize the recursive search should solve a slightly more general problem than the original prompt. That generalized problem becomes the state.

In the text segmentation example, you start with a clean recursive idea: choose the first word, and recursively segment the rest of the string. If any split works, return `true`.

At first that sounds like recursion on explicit substrings. Erickson then rewrites it in index form: instead of passing around a suffix `A[i..n]`, treat the original string as global and describe each subproblem only by the starting index `i`. It looks like a small notation change, but it is a useful habit.

The resulting state is:

$$
\text{Splittable}(i) = \text{whether the suffix } A[i..n] \text{ can be segmented into words}
$$

with base case $\text{Splittable}(n+1)=\text{true}$ and recurrence

$$
\text{Splittable}(i) =
\bigvee_{j=i}^{n}
\Big( \text{IsWord}(i,j) \wedge \text{Splittable}(j+1) \Big)
$$

Once you write it this way, the repeated work becomes obvious. Different branches of the recursion keep asking the exact same suffix question. "Can I segment starting at index 17?" means the same thing no matter *how* you got to index 17.

Because the state is so compact, the algorithm will evaluate the same states over and over. If you draw the recursive calls, they stop looking like a pure tree pretty quickly; branches keep merging into a graph of overlapping states.

That is the step from backtracking to DP. First identify the decision structure. Then shrink the state to the smallest summary that matters. Once many paths land on the same state, you stop recomputing and start saving answers.

### What to remember

- **Backtracking makes one decision per recursive call.** The output often looks like a sequence of decisions.
- **`n`-queens is the standard picture.** Each node is a legal partial board, and the algorithm does DFS over that implicit tree.
- **The main design question is what past information you need to keep.** That is usually the recursive state.
- **Index-based formulations matter.** Parameters like `i`, `j`, or remaining target make repeated states easier to see.
- **The first backtracking solution is often a draft.** Its value is showing you where the overlapping subproblems are.

## Dynamic Programming

After looking at Backtracking this way, Dynamic Programming feels less like a separate trick and more like the next step. Start with a recursive search over choices, shrink the state to what actually matters, and then notice that many branches keep asking for the same state.

That is the main difference from Divide & Conquer. In D&C, the recursive calls are **independent**. In DP, the subproblems **overlap**. The search is still there in spirit, but instead of exploring a huge tree of repeated work, we collapse equivalent calls into a Directed Acyclic Graph (DAG) of states and solve each state once.

The advantage of DP is not that it avoids recursion conceptually. It avoids recomputation. Once the subproblem definition is right, the rest is bookkeeping: store the answer for each state and evaluate states in an order that respects their dependencies.

The rest of the section groups common DP families by the shape of their state graph, the transition they use, and where the final answer lives. *Click through the patterns and flavors in the visualizers below to see how the dependency structure changes.*

### Explaining a DP Solution

There is a real difference between *discovering* a DP solution and *writing one up*. As in the Divide & Conquer section, jumping straight to code can hide shaky reasoning. The same broad "What, How, Why, How fast" template still works, but the emphasis shifts toward state definitions and transitions.

**1. What: Problem Specification.** Define the problem you're trying to solve recursively in coherent and precise English. Translate the colloquial statement into sets, sequences, or mathematical terms. State clearly what subset of the input it operates on and any constraints. 

**2. How: State and Transition.**
- **State.** Define the meaning of a table entry like $T(i)$ or $T(i,j)$ in plain English. A precise state definition usually does most of the conceptual work. If this part is vague, the recurrence is usually vague too.
- **Recurrence.** Write the rule that computes one state from smaller ones in a precise mathematical or algorithmic form. Good transitions make the dependency graph obvious: who depends on whom, and what has to be known first. Base cases and variable bounds should be included.

**3. Why: Justification of Correctness.** Explain the logical basis for the recurrence transitions. This acts as an informal inductive proof that your subproblem choices and state transitions correctly build the optimal solution.

**4. How Fast: Efficiency Checklist.** For a complete analysis, it helps to separate four questions:
1. **State Space:** How many distinct subproblems exist?
2. **Transition Time:** How much work is required to compute one subproblem?
3. **Extraction:** Where is the final answer located in the table or DAG?
4. **Extraction Time:** How much additional work is required to read or reconstruct that answer?

> [!note]
> **On Extraction:** Just finding the optimal numeric value is often not enough. If a problem asks for the optimal sequence or path, you must trace back through the table/DAG to reconstruct your choices. This typically takes $O(n)$ or time proportional to the path length.

> [!tip]
> **Implementation Rules of Thumb:** DP is often clearest when it is presented as a bottom-up table fill over immutable inputs. Base cases can start at index $0$ or even $-1$ if that keeps the recurrence well-defined at the boundaries.

### Problem-Solving Heuristics

Before a polished explanation can be written, the recursive idea usually has to be sorted out on scratch paper. Erickson's split is useful here: first get the recursive idea right, then turn that into a table or DAG. The following questions tend to help in that order:

**Stage 1: Formulate the Problem Recursively (The Hard Part)**
1. **What is the shape of the state space?** Is the input a 1D array, a pair of strings, items with weights, an interval, a tree, or a set of elements?
2. **What are you optimizing?** Max/min value? Count of ways? Length of a subsequence?
3. **What constraint distinguishes the flavor?** Must it be contiguous (Kadane) vs. no-adjacent (Robber) vs. any subsequence (LIS)? Match-based (LCS) vs. cost-based (Edit Dist)? Items used once (0/1) vs. unlimited (Unbounded)?
4. **Write the subproblem in words first.** A good recurrence usually falls out of a precise English description of the subproblem.
5. **Always define base cases.** Ensure every referenced table entry is well-defined. If $T(i)$ references $T(i-2)$, define $T(0)$ and possibly $T(-1)$.

**Stage 2: Build Solutions Bottom-Up (The Mechanical Part)**
6. **Identify the state space.** What are all the different ways the recursive algorithm can call itself? This usually comes from the smallest summary of past decisions. The number of combinations dictates the dimensions of the DP table.
7. **Identify dependencies.** Except for base cases, what dictates a cell's value? (e.g., $T(i, j)$ might depend on $T(i-1, j-1)$). This is where the search tree turns into a DAG of dependencies.
8. **Find a good evaluation order.** You must compute subproblems *before* the things that depend on them. The dependencies you just identified dictate whether you loop left-to-right, right-to-left, bottom-up, or diagonally. Provide bounds for all variables.
9. **Analyze time.** The total running time is simply the number of distinct subproblems (table size) multiplied by the time to compute a single subproblem.

### Pattern A: Linear (1D Chain)

Linear patterns are the simplest kind of Dynamic Programming. The state space is a 1D sequence, and the dependency graph is a straight chain from left to right. These problems usually feel like "walk through the array once, but keep the right summary of the past." The main question is: *How does the current element connect to the best solution so far?*

<div class="viz-shell">
  <DPDagVisualizer client:visible pattern="A" />
</div>

The structure is a 1D array. The flavor depends on how far back you look and what each state stores.

#### Flavor 1: "Look-Back-One" (Contiguous / Kadane's Style)

**1. What: Problem Specification**
Given an array `A[1..n]`, find the maximum sum of any contiguous subarray.

**2. How: Algorithm Description**
**Concept:** _Either extend the previous subproblem or start over at the current element. The current element must be included._

**(a) Subproblem:** T(i) = maximum sum of a contiguous subarray of A[1..i] which includes A[i].

**(b) Recurrence:**

**Base Cases:**

$$
T(1) = A[1]
$$

**Recursive Step:**

$$
\begin{aligned}
T(i) &= \max( A[i],  T(i-1) + A[i] ) \\
     &\quad \text{for } 2 \le i \le n
\end{aligned}
$$

> [!note]
> _Interpretation: either start a new subarray at A[i], or extend the best subarray ending at A[i-1]._

**3. Why: Justification of Correctness**
Any optimal contiguous subarray ending at $i$ either starts at $A[i]$ or extends the best contiguous subarray ending at $i-1$. Checking both cases gives the right value for $T(i)$.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n)`
- **(2) Fill time:** `O(n)`
- **(3) Return:** `max{ T(i) : 1 ≤ i ≤ n }`
- **(4) Return time:** `O(n)`

---

#### Flavor 2: "Look-Back-Two" (Skip / House Robber Style)

**1. What: Problem Specification**
Given an array of values `A[1..n]`, find the maximum sum obtainable such that no two adjacent elements are selected.

**2. How: Algorithm Description**
**Concept:** _Adjacent elements cannot both be selected. Either take the current element and skip the previous one, or leave it and keep the previous optimum._

**(a) Subproblem:** T(i) = maximum sum obtainable from A[1..i] (A[i] may or may not be selected).

**(b) Recurrence:**

**Base Cases:**

$$
\begin{align*}
T(0) &= 0 \\
T(1) &= A[1]
\end{align*}
$$

**Recursive Step:**

$$
\begin{aligned}
T(i) &= \max( T(i-1),  T(i-2) + A[i] ) \\
     &\quad \text{for } 2 \le i \le n
\end{aligned}
$$

> [!note]
> _T(i-1) = skip A[i]; T(i-2) + A[i] = take A[i] and skip A[i-1]._

**3. Why: Justification of Correctness**
At step $i$, either take $A[i]$ and pair it with the best solution ending at $i-2$, or skip $A[i]$ and keep the best solution through $i-1$. Those are the only legal options, so the maximum of the two is correct.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n)`
- **(2) Fill time:** `O(n)`
- **(3) Return:** `T(n)`
- **(4) Return time:** `O(1)`

---

#### Flavor 3: "Look-Back-All" (Subsequence / LIS Style)

**1. What: Problem Specification**
Given an array `A[1..n]`, find the length of the longest increasing subsequence. The sequence need not be contiguous.

**2. How: Algorithm Description**
**Concept:** _The sequence need not be contiguous. Scan all earlier valid predecessors and extend the best one. The current element must be included._

**(a) Subproblem:** T(i) = length of the longest increasing subsequence of A[1..i] which includes A[i].

**(b) Recurrence:**

**Base Cases:**

$$
T(1) = 1
$$

**Recursive Step:**

$$
\begin{aligned}
T(i) &= 1 + \max \big( \{0\} \cup \{ T(j) : \\
     &\quad 1 \le j < i,\; A[j] < A[i] \} \big) \\
     &\quad \text{for } 2 \le i \le n
\end{aligned}
$$

> [!note]
> _If no j satisfies A[j] < A[i], the max is taken over {0} (set containing zero), so T(i) = 1._

**3. Why: Justification of Correctness**
To compute the longest increasing subsequence ending at $i$, check every earlier index $j$ with $A[j] < A[i]$ and extend the best one. If no such $j$ exists, the subsequence starts at $i$ and has length 1.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n)`
- **(2) Fill time:** `O(n²)`
- **(3) Return:** `max{ T(i) : 1 ≤ i ≤ n }`
- **(4) Return time:** `O(n)`

### Pattern B: Grid (2D / Strings)

Grid patterns move the state space into two dimensions. The dependencies usually come from the cell above, the cell to the left, or the diagonal. They show up when you are aligning two sequences or when each move through a matrix depends on a small local neighborhood.

<div class="viz-shell">
  <DPDagVisualizer client:visible pattern="B" />
</div>

The local neighborhood is usually `(i-1, j)`, `(i, j-1)`, and `(i-1, j-1)`. The flavor depends on whether you are matching symbols or accumulating cost.

#### Flavor 1: "Match or Drop" (Longest Common Subsequence)

**1. What: Problem Specification**
Given two strings `A[1..n]` and `B[1..m]`, find the length of their longest common subsequence.

**2. How: Algorithm Description**
**Concept:** _If elements match, gain 1 and consume both. If not, drop one element from either sequence and take the best._

**(a) Subproblem:** T(i,j) = length of the LCS of A[1..i] and B[1..j].

**(b) Recurrence:**

**Base Cases:**

$$
\begin{align*}
T(0,j) &= 0 \quad \text{for } 0 \le j \le m \\
T(i,0) &= 0 \quad \text{for } 0 \le i \le n
\end{align*}
$$

**Recursive Step:**

$$
T(i,j) =
\begin{aligned}
&\begin{cases}
T(i-1,j-1) + 1 & \text{if } A[i] = B[j] \\
\max( T(i-1,j),  T(i,j-1) ) & \text{if } A[i] \ne B[j]
\end{cases} \\
&\quad \text{for } 1 \le i \le n, 1 \le j \le m
\end{aligned}
$$

**3. Why: Justification of Correctness**
If the current elements match ($A[i] = B[j]$), they can extend the optimal common subsequence, so we gain 1 and consume both. If they do not match, we must drop one element from either sequence and take the best of the two options. This evaluates all possible alignments for a common subsequence.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n·m)`
- **(2) Fill time:** `O(n·m)`
- **(3) Return:** `T(n,m)`
- **(4) Return time:** `O(1)`

---

#### Flavor 2: "Penalty Cost" (Edit Distance)

**1. What: Problem Specification**
Given two strings `A[1..n]` and `B[1..m]`, find the minimum number of edits (insertions, deletions, substitutions) required to transform A into B.

**2. How: Algorithm Description**
**Concept:** _Transform A into B. Every mismatch incurs a cost; take the minimum of substitution, deletion, or insertion._

**(a) Subproblem:** T(i,j) = minimum edit distance to transform A[1..i] into B[1..j].

**(b) Recurrence:**

**Base Cases:**

$$
\begin{align*}
T(0,j) &= j \quad \text{for } 0 \le j \le m \quad (\text{insert } j \text{ characters}) \\
T(i,0) &= i \quad \text{for } 0 \le i \le n \quad (\text{delete } i \text{ characters})
\end{align*}
$$

**Recursive Step:**

$$
T(i,j) =
\begin{cases}
T(i-1,j-1) & \text{if } A[i] = B[j] \\
1 + \min\big( T(i-1,j-1), \\
\quad T(i-1,j), T(i,j-1) \big) & \text{if } A[i] \ne B[j]
\end{cases}
$$

**3. Why: Justification of Correctness**
To transform A into B, we process from the ends. If the current characters match, no cost is incurred, and we reduce both strings. If they mismatch, we must pay a penalty of 1 and choose the cheapest operation: substitution ($i-1, j-1$), deletion ($i-1, j$), or insertion ($i, j-1$). This systematically minimizes total edits over all alignments.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n·m)`
- **(2) Fill time:** `O(n·m)`
- **(3) Return:** `T(n,m)`
- **(4) Return time:** `O(1)`

---

#### Flavor 3: "Path Accumulation" (Min Path Sum)

**1. What: Problem Specification**
Given a grid of costs `C[1..n][1..m]`, you can only move right or down. Find the minimum cost to travel from the top-left cell to the bottom-right cell.

**2. How: Algorithm Description**
**Concept:** _Move through a grid and accumulate cost. The only constraint is that moves go right or down._

**(a) Subproblem:** T(i,j) = minimum cost to reach cell (i,j) from cell (1,1).

**(b) Recurrence:**

**Base Cases:**

$$
\begin{align*}
T(1,1) &= C[1][1] \\
T(i,1) &= T(i-1,1) + C[i][1] \quad \text{for } 2 \le i \le n \\
T(1,j) &= T(1,j-1) + C[1][j] \quad \text{for } 2 \le j \le m
\end{align*}
$$

**Recursive Step:**

$$
\begin{aligned}
T(i,j) &= C[i][j] + \min( T(i-1,j), T(i,j-1) ) \\
       &\quad \text{for } 2 \le i \le n, 2 \le j \le m
\end{aligned}
$$

**3. Why: Justification of Correctness**
Because movement is restricted to right and down, any path to $(i, j)$ must come from either $(i-1, j)$ or $(i, j-1)$. So the cheapest path to $(i, j)$ is the cheaper of those two predecessors plus $C[i][j]$.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n·m)`
- **(2) Fill time:** `O(n·m)`
- **(3) Return:** `T(n,m)`
- **(4) Return time:** `O(1)`

---

### Pattern C: Knapsack

Knapsack patterns are constrained optimization problems: you have a limited capacity and a list of items. The state space is usually 2D, with one dimension for how many items you have considered and one for the remaining capacity. The important distinction is whether items can be reused.

<div class="viz-shell">
  <DPDagVisualizer client:visible pattern="C" />
</div>

The dependency graph jumps left across the capacity dimension whenever you take an item. The flavor changes based on whether items are unique, reusable, or being counted.

#### Flavor 1: "Take It or Leave It" (0/1 Knapsack)

**1. What: Problem Specification**
Given `n` items with weights $w_i$ and values $v_i$, and a total weight capacity $W$, find the maximum value of a subset of items that fits in the capacity. Each item can be used at most once.

**2. How: Algorithm Description**
**Concept:** _Each item can only be used once. You transition from the previous row (i-1)._

**(a) Subproblem:** T(i,w) = maximum value achievable using items from {1..i} with capacity w.

**(b) Recurrence:**

**Base Cases:**

$$
T(0,w) = 0 \quad \text{for } 0 \le w \le W
$$

**Recursive Step:**

$$
\begin{aligned}
T(i,w) &= \begin{cases}
T(i-1,w) & \text{if } w_i > w \\
\max( T(i-1,w),  T(i-1, w-w_i) + v_i ) & \text{if } w_i \le w
\end{cases} \\
&\quad \text{for } 1 \le i \le n, 0 \le w \le W
\end{aligned}
$$

> [!note]
> _wᵢ and vᵢ are the weight and value of item i._

**3. Why: Justification of Correctness**
Each item forces a binary choice: either include it or exclude it. If the item's weight exceeds the available capacity $w$, it must be excluded. If it fits, we take the maximum between excluding it ($T(i-1, w)$) or including it ($T(i-1, w-w_i) + v_i$). Both choices transition from the previous row ($i-1$), ensuring each item is used at most once.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n·W)`
- **(2) Fill time:** `O(n·W)`
- **(3) Return:** `T(n,W)`
- **(4) Return time:** `O(1)`

---

#### Flavor 2: "Infinite Reuse" (Unbounded Knapsack / Min Coin Change)

**1. What: Problem Specification**
Given a target amount $W$ and $n$ coin denominations, find the minimum number of coins needed to make change. Each denomination can be used infinitely many times.

**2. How: Algorithm Description**
**Concept:** _If you take an item, you can take it again. That means you stay in the current row instead of moving to the previous one._

**(a) Subproblem:** T(j) = minimum number of coins needed to make change for amount j, using coins from the given denominations.

**(b) Recurrence:**

**Base Cases:**

$$
\begin{align*}
T(0) &= 0 \\
T(j) &= \infty \quad \text{if no coin } \le j
\end{align*}
$$

**Recursive Step:**

$$
\begin{aligned}
T(j) &= 1 + \min_{ \substack{1 \le i \le n\\c_i \le j} } \{ T(j - c_i) \} \\
     &\quad \text{for } 1 \le j \le W
\end{aligned}
$$

> [!note]
> _Can also be written as 2D: T(i,w) = min( T(i-1,w), T(i, w-cᵢ) + 1 ), where row i means "using coins 1..i" and staying in row i allows reuse._

**3. Why: Justification of Correctness**
If amount $j$ can be formed, the last coin used is some $c_i \le j$. So the answer is 1 plus the best solution for $j-c_i$, minimized over all valid coin choices.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(W)`
- **(2) Fill time:** `O(n·W)`
- **(3) Return:** `T(W)`
- **(4) Return time:** `O(1)`

---

#### Flavor 3: "Counting Combinations" (Subset Sum / Coin Change 2)

**1. What: Problem Specification**
Given a target amount $W$ and $n$ coin denominations, find the total number of ways to make change.

**2. How: Algorithm Description**
**Concept:** _You are not maximizing value here. You are counting ways, so the transition adds cases instead of taking a max._

**(a) Subproblem:** T(i,w) = number of ways to form amount w using coins from {1..i}.

**(b) Recurrence:**

**Base Cases:**

$$
\begin{align*}
T(0,0) &= 1 \\
T(0,w) &= 0 \quad \text{for } 1 \le w \le W
\end{align*}
$$

**Recursive Step:**

$$
\begin{aligned}
T(i,w) &= \begin{cases}
T(i-1,w) & \text{if } c_i > w \\
T(i-1,w) + T(i, w-c_i) & \text{if } c_i \le w
\end{cases} \\
&\quad \text{for } 1 \le i \le n, 0 \le w \le W
\end{aligned}
$$

> [!note]
> _Staying in row i (for the cᵢ ≤ w case) allows reuse of coin i. Moving to row i-1 only would give 0/1 counting._

**3. Why: Justification of Correctness**
The number of ways to form amount $w$ with coins $1..i$ is the sum of two disjoint cases: ways that do not use $c_i$, and ways that use at least one $c_i$.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n·W)`
- **(2) Fill time:** `O(n·W)`
- **(3) Return:** `T(n,W)`
- **(4) Return time:** `O(1)`

---

### Pattern D: Interval (Pyramid)

Interval patterns work on a continuous span `[i..j]`. Instead of building one index at a time, you solve bigger intervals from smaller ones. The main variation is how the interval gets reduced: split at some `k`, shrink from the edges, or reason about the last action inside the interval.

<div class="viz-shell">
  <DPDagVisualizer client:visible pattern="D" />
</div>

The structure always combines sub-intervals. The flavor depends on whether you split at an arbitrary `k`, shrink from the edges, or reason from the last action.

#### Flavor 1: "The K-Split" (Matrix Chain Multiplication / Optimal BST)

**1. What: Problem Specification**
Given a sequence of $n$ matrices, find the most efficient way (minimum scalar multiplications) to multiply these matrices together. Matrix $A_i$ has dimensions $p[i-1] \times p[i]$.

**2. How: Algorithm Description**
**Concept:** _The cost of interval (i,j) depends on splitting it into (i,k) and (k+1,j) plus the cost to merge._

**(a) Subproblem:** T(i,j) = minimum cost to multiply matrices Aᵢ through Aⱼ.

**(b) Recurrence:**

**Base Cases:**

$$
T(i,i) = 0 \quad \text{for } 1 \le i \le n
$$

**Recursive Step:**

$$
\begin{aligned}
T(i,j) &= \min_{ i \le k \le j-1} \{ T(i,k) + T(k+1,j) + p[i-1] \cdot p[k] \cdot p[j] \} \\
       &\quad \text{for } 1 \le i < j \le n
\end{aligned}
$$

> [!note]
> _Fill order: by increasing interval length ℓ = j - i, from ℓ=1 up to ℓ=n-1._

**3. Why: Justification of Correctness**
Any parenthesization of the product $A_i \dots A_j$ must split the product at some matrix $A_k$. The minimum cost is the cost to compute the left half ($T(i,k)$), plus the right half ($T(k+1,j)$), plus the cost to multiply those two resulting matrices ($p[i-1]p[k]p[j]$). By iterating over all possible split points $k$, we find the optimal cost for the interval.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n²)`
- **(2) Fill time:** `O(n³)` (due to testing $O(n)$ splits $k$ per subproblem)
- **(3) Return:** `T(1,n)`
- **(4) Return time:** `O(1)`

---

#### Flavor 2: "Shrink from Edges" (Longest Palindromic Subsequence)

**1. What: Problem Specification**
Given a string $A[1..n]$, find the length of its longest palindromic subsequence.

**2. How: Algorithm Description**
**Concept:** _No k-loop needed. Compare the two outermost elements. If they match, the interval shrinks by 2. If not, shrink by 1 from either side._

**(a) Subproblem:** T(i,j) = length of the longest palindromic subsequence of A[i..j].

**(b) Recurrence:**

**Base Cases:**

$$
T(i,i) = 1 \quad \text{for } 1 \le i \le n
$$

**Recursive Step:**

$$
\begin{aligned}
T(i,j) &= \begin{cases}
T(i+1,j-1) + 2 & \text{if } A[i] = A[j] \\
\max( T(i+1,j),  T(i,j-1) ) & \text{if } A[i] \ne A[j]
\end{cases} \\
&\quad \text{for } 1 \le i < j \le n
\end{aligned}
$$

> [!note]
> _Base case T(i,i+1): if A[i]=A[i+1] then 2, else 1. This is handled by the recurrence if you define T(i,i-1)=0 for empty intervals._

**3. Why: Justification of Correctness**
We do not need an arbitrary split point here. If the outer characters match, they can be the ends of the palindrome and contribute 2. If they do not, the optimal subsequence must drop either the left end or the right end.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n²)`
- **(2) Fill time:** `O(n²)`
- **(3) Return:** `T(1,n)`
- **(4) Return time:** `O(1)`

---

#### Flavor 3: "The Last Action" (Burst Balloons)

**1. What: Problem Specification**
Given $n$ balloons with values $A[1..n]$, popping balloon $k$ yields $A[k-1] \cdot A[k] \cdot A[k+1]$ coins. Find the maximum coins obtained by bursting all balloons. Balloons $0$ and $n+1$ are virtual with value $1$.

**2. How: Algorithm Description**
**Concept:** _Treat `k` as the last balloon popped in `(i,j)`. That makes the left and right subproblems independent._

**(a) Subproblem:** T(i,j) = maximum coins from bursting all balloons in the open interval (i,j), where i and j are unpopped boundary balloons.

**(b) Recurrence:**

**Base Cases:**

$$
T(i,j) = 0 \quad \text{if } j = i+1 \quad (\text{no balloons between})
$$

**Recursive Step:**

$$
\begin{aligned}
T(i,j) &= \max_{i < k < j} \{ T(i,k) + T(k,j) + A[i] \cdot A[k] \cdot A[j]\} \\
       &\quad \text{for } 0 \le i < j \le n+1
\end{aligned}
$$

> [!note]
> _Boundaries A[0] and A[n+1] are sentinel values = 1. The key insight: choosing k last means the left/right subproblems don't interfere._

**3. Why: Justification of Correctness**
Pick the *last* balloon popped in the interval, not the first. Once $k$ is last, the left interval $(i,k)$ and right interval $(k,j)$ stop interfering with each other, so we can optimize them independently and maximize over all choices of $k$.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n²)`
- **(2) Fill time:** `O(n³)`
- **(3) Return:** `T(0, n+1)`
- **(4) Return time:** `O(1)`

---

### Pattern E: Tree DP

Tree DP applies the same idea to hierarchical data. A subtree's answer depends only on its children, so post-order traversal usually gives the right evaluation order. You often need multiple states per node.

<div class="viz-shell">
  <DPDagVisualizer client:visible pattern="E" />
</div>

Tree DP often uses multiple states per node to handle inclusion or exclusion choices. Fill order is post-order: leaves first, root last.

#### Flavor 1: "Independent Set" (Max Weight Independent Set / House Robber III)

**1. What: Problem Specification**
Given a tree where each node $u$ has a weight $w(u)$, find the maximum weight of an independent set (a set of nodes where no two nodes are connected by an edge).

**2. How: Algorithm Description**
**Concept:** _You cannot select a node and its direct children. Two states per node: exclude or include._

**(a) Subproblem:** T(u,0) = max weight of an independent set in the subtree rooted at u, excluding u. T(u,1) = same, including u.

**(b) Recurrence:**

**Base Cases:**

$$
\begin{align*}
T(u,0) &= 0 \\
T(u,1) &= w(u)
\end{align*}
\quad \text{for leaf node } u
$$

**Recursive Step:**

$$
\begin{align*}
T(u,0) &= \sum \max( T(v,0), T(v,1) ) \\
       &\quad \text{over all children } v \text{ of } u \\
T(u,1) &= w(u) + \sum T(v,0) \\
       &\quad \text{over all children } v \text{ of } u
\end{align*}
$$

> [!note]
> _When u is excluded, each child is free to be included or not. When u is included, all children must be excluded._

**3. Why: Justification of Correctness**
If $u$ is included, its children must be excluded. If $u$ is excluded, each child is free to be included or excluded. Tracking both states at every node captures all legal cases.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n)  (2 states per node)`
- **(2) Fill time:** `O(n)`
- **(3) Return:** `max( T(root,0), T(root,1) )`
- **(4) Return time:** `O(1)`

---

#### Flavor 2: "Path Tracking" (Diameter of a Tree)

**1. What: Problem Specification**
Given a tree, find the diameter: the length of the longest path between any two nodes.

**2. How: Algorithm Description**
**Concept:** _The longest path may pass through a node. Track the best downward path from `u`, then combine the top two child paths when needed._

**(a) Subproblem:** T(u) = length of the longest path starting at u and going downward into the subtree of u.

**(b) Recurrence:**

**Base Cases:**

$$
T(u) = 1 \quad \text{for leaf node } u
$$

**Recursive Step:**

$$
\begin{aligned}
T(u) &= 1 + \max \{ T(v) : v \text{ is a child of } u \} \\
     &\quad (\text{longest arm in nodes})
\end{aligned}
$$

To extract the exact answer for the diameter:

$$
\begin{aligned}
&\text{Diameter through } u = T(v_1) + T(v_2) + 1 \\
&\quad (\text{where } v_1, v_2 \text{ are children with largest } T)
\end{aligned}
$$

$$
\text{Answer} =
\begin{cases}
\max_u \{ \text{diameter through } u \} - 1 & (\text{if asking for edges}) \\
\max_u \{ \text{diameter through } u \} & (\text{if asking for nodes})
\end{cases}
$$

> [!note]
> _If u has ≤1 child, diameter through u is just T(u). Track the global max during the fill pass._

**3. Why: Justification of Correctness**
The longest path either passes through some node $u$ or stays entirely inside one subtree. Once the downward values $T(v)$ are known for the children of $u$, the best path through $u$ comes from combining the two largest child branches. Taking the maximum over all nodes gives the diameter.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n)`
- **(2) Fill time:** `O(n)`
- **(3) Return:** `max over all u of (top two T(child) values summed)`
- **(4) Return time:** `O(n)`

---

### Pattern F: Bitmask (Subset DP)

Bitmask DP works on sets instead of plain sequences. When $N$ is small, usually around $20$ or less, an exponential $O(2^N)$ state space is often acceptable. A bitmask stores which elements are already used or visited, so subsets become direct state variables.

<div class="viz-shell">
  <DPDagVisualizer client:visible pattern="F" />
</div>

Each subset is encoded as an integer whose bits say which elements are present. The flavor changes based on whether the state also needs an endpoint.

#### Flavor 1: "Path Building" (Traveling Salesperson Problem)

**1. What: Problem Specification**
Given $n$ cities and the distances between each pair, find the minimum distance to visit every city exactly once and end at a specific city $i$.

**2. How: Algorithm Description**
**Concept:** _You need both the visited set `S` and the current city `i`, because the next move depends on both._

**(a) Subproblem:** T(S,i) = minimum cost to visit exactly the cities in set S, starting from city 1 and ending at city i, where i ∈ S.

**(b) Recurrence:**

**Base Cases:**

$$
T(\{1\}, 1) = 0
$$

**Recursive Step:**

$$
\begin{aligned}
T(S, i) &= \min_{j \in S\setminus\{i\}} \{ T(S\setminus\{i\}, j) + d(j,i) \} \\
        &\quad \text{for all } S \subseteq \{1..n\} \text{ with } 1 \in S, i \in S, i \ne 1
\end{aligned}
$$

> [!note]
> _S\{i} means the set S with city i removed. Fill order: by increasing |S|. d(j,i) = distance from j to i._

**3. Why: Justification of Correctness**
Any optimal path that visits the set $S$ and ends at $i$ must come from some earlier city $j \in S \setminus \{i\}$. So we try every possible predecessor $j$ and take the minimum.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(2ⁿ · n)`
- **(2) Fill time:** `O(2ⁿ · n²)`
- **(3) Return:** `min{ T({1..n}, i) + d(i,1) : 2 ≤ i ≤ n }`
- **(4) Return time:** `O(n)`

---

#### Flavor 2: "Subset Partitioning" (Minimum Cost to Partition into Groups)

**1. What: Problem Specification**
Given a set of $n$ elements and a cost function for forming a group out of any subset, find the optimal partitioning of all elements into groups to minimize the total cost.

**2. How: Algorithm Description**
**Concept:** _Order and endpoints do not matter here. Split the full set into one chosen group `S'` plus the rest, and try all submasks._

**(a) Subproblem:** T(S) = minimum cost to optimally partition the elements in set S into groups.

**(b) Recurrence:**

**Base Cases:**

$$
T(\emptyset) = 0
$$

**Recursive Step:**

$$
\begin{aligned}
T(S) &= \min_{\substack{S' \subseteq S \\ S' \ne \emptyset}} \{ T(S \setminus S') + \text{cost}(S') \} \\
     &\quad \text{for all non-empty } S \subseteq \{1..n\}
\end{aligned}
$$

> [!note]
> _The 3ⁿ runtime comes from iterating over all submasks of all masks. Each element is in S', in S\S', or not in S, giving three choices per element._

**3. Why: Justification of Correctness**
Any partition of $S$ must contain some non-empty group $S'$. Once that group is chosen, the rest of the problem is just the optimal partition of $S \setminus S'$, so checking all submasks covers every valid partition.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(2ⁿ)`
- **(2) Fill time:** `O(3ⁿ)` (due to iterating over all submasks of all masks)
- **(3) Return:** `T({1..n})`
- **(4) Return time:** `O(1)`

### What to remember

- **Common shapes**: 1D chain, 2D grid, knapsack, interval, tree, and bitmask.
- **Subproblem definition matters**: `T(i)` or `T(i,j)` should map cleanly to the final goal in plain English.
- **A good explanation has a few moving parts**: problem statement, state, transition, and complexity.
- **Fill order matters**: whichever representation you use, dependencies have to be available before you try to use them.
