## Divide & Conquer

Use this to map a recurrence to Master Theorem behavior and see which level dominates work as parameters change.

### The 4-Step Solution Format

The course encourages structuring problem-solving into distinct sections rather than diving straight into code. This approach builds a habit of thinking algorithmically and mathematically. By combining the course requirements with Jeff Erickson's "What, How, Why, How fast" framework, we get a robust template for explaining algorithms. _Note: Correctness is never assumed from the algorithm steps; it must be explicitly argued._

- **1. What: Problem Specification:** Explicitly define the problem the algorithm solves. Include the type and meaning of inputs, exactly what the output should be, and state any hidden assumptions.
- **2. How: Algorithm Description:** Explain _how_ your algorithm solves the problem using a plain-English narrative. You can use paragraphs or high-level bullet points, but **pseudocode is strictly forbidden**.
  - Avoid writing code disguised as words (e.g., translating a script line-by-line). Deeply nested bullet points are usually a red flag.
  - Walk through every major step, ending with the final return statement.
  - If you are adapting a known algorithm, explicitly highlight the modifications you made.
- **3. Why: Justification of Correctness:** Explain _why_ your algorithm works.
  - Write a narrative, informal justification (no formal mathematical or inductive proofs are required).
  - If you modified a known algorithm, you must explain why your specific tweaks are correct and successfully solve the problem.
- **4. How Fast: Runtime Analysis:** Analyze your algorithm using worst-case **Big-O notation**.
  - You can skip analyzing $O(1)$ operations.
  - You may assert the standard runtime of unmodified "black-box" algorithms without extra proof.
  - If you modify a black box, you must justify how those changes impact the runtime—even if the overall Big-O doesn't change.
  - Always conclude with a single, fully simplified final runtime in Big-O notation.

### Guide to finding a D&C solution

Follow this workflow to brainstorm and write your solution:

1. **Identify the Pattern:** Determine if your problem maps to a standard algorithm.
   - _Sorted data_ usually points to Binary Search.
   - _Unsorted data_ usually points to Merge Sort or FastSelect.
   - _Polynomials, convolution, or multiplication_ usually point to Fast Fourier Transform (FFT). For integer multiplication specifically, Karatsuba's algorithm is a classic approach ($O(n^{\log_2 3})$).
2. **Define Modifications (If Any):** Decide if you can use the known algorithm "as is" or if it needs tweaking. If you tweak it, clearly outline what steps, inputs, or outputs are changing.
3. **Draft the Steps:** Write out the algorithm's execution in clear words. Make sure to define your base cases (if needed) and ensure your algorithm ultimately returns exactly what the prompt asks for.
4. **Draft the Proof of Correctness:** Write out your informal proof. To build a strong justification, answer questions like:
   - _Why is this specific black box suited for this input data?_
   - _Why did you make those specific modifications?_
   - _Why are we guaranteed that the problem space shrinks on each round?_
   - _Why are the base cases logically sound?_
   - _What is the logic behind branching left or right on each recursive call?_
5. **Calculate the Runtime:** Step through your algorithm to build your final worst-case Big-O analysis.

### Master Theorem

For some constants $a > 0$, $b > 1$, and $d \geq 0$,
where:
$a$ - number of subproblems
$b$ - is the factor by which the input size shrinks
$d$ - is the exponent of the work done at each level of recursion (so combining costs $O(n^d)$)

$$
T(n) = \underbrace{aT (n/b)}_{\text{recursive branching}} + \underbrace{O(n^d)}_{\text{local work (root)}}
$$

$$
T(n) =
\begin{cases}
\Theta(n^{\log_b(a)}) &\text{if } n^{\log_b(a)} > n^d \quad\text{ ``leaf heavy"} \\
\Theta(n^d \log(n)) &\text{if } n^{\log_b(a)} = n^d \quad\text{ ``balanced"}\\
\Theta(n^d) &\text{if } n^{\log_b(a)} < n^d \quad\text{ ``root heavy"} \\
\end{cases}
$$

- **Case 1 (Leaf-Heavy):** $n^{\log_b(a)} > n^d$. The recursive branching explodes, meaning the vast majority of the work happens at the bottom of the tree. The leaves win, so the bound is strictly the number of leaves: $\Theta(n^{\log_b(a)})$.

- **Case 2 (Perfectly Balanced):** $n^{\log_b(a)} = n^d$. It is a tie. The local work at the root matches the work at the leaves, and the work across every internal level of the tree is identical. Because the work per level is constant, you just multiply the root work by the number of levels ($\log n$). The bound is: $\Theta(n^d\log n)$. _(Note: The "extended" Case 2 handles $f(n) = \Theta(n^{\log_b a} \log^k n)$, resulting in $\Theta(n^{\log_b a} \log^{k+1} n)$)._
- **Case 3 (Root-Heavy):** $n^{\log_b(a)} < n^d$. The local work dwarfs the branching. The work shrinks geometrically as you go down the tree, meaning the total time is dominated by the massive chunk of work done at the very beginning. The root wins, so the bound is strictly the root work: $\Theta(n^d)$.

### Work per level

1. **Number of nodes:** At level $k$, the tree has branched $k$ times, so there are $a^k$ nodes.
2. **Size of each subproblem:** The problem size $n$ has been divided by $b$ exactly $k$ times, making the input size for each node $n/b^k$.
3. **Work per node:** Since the local work is bounded by $O(\text{size}^d)$, the work at a single node on level $k$ is $(\frac{n}{b^k})^d$, which simplifies to $\frac{n^d}{b^{kd}}$.

If you multiply the **number of nodes** by the **work per node**, you get the total work for the entire $k$-th level:

$$
a^k \times \frac{n^d}{b^{kd}}
$$

Pull out the $n^d$ (which is the work at the root), and group the $k$ exponents together, and you get the formula for the total work done at the $k$-th level of the tree:

$$
O(n^d) \times \left(\frac{a}{b^d}\right)^k
$$

The Master Theorem uses $\log_b(a)$ instead of just comparing $a$ to $b^d$. It is just a logarithmic algebraic rewrite of the exact same inequality!

If you take $a=b^d$ and take the $\log_b$ of both sides, you get:

$$
\log_b(a)=\log_b(b^d)
$$

$$
\log_b(a)=d
$$

So, comparing $\log_b(a)$ to $d$ is mathematically identical to comparing the branching factor $a$ to the work-shrinkage factor $b^d$.

### Example Walkthrough: Majority Element

Let's walk through a concrete example using the 4-step format.

**1. What: Problem Specification**
An array `A[1…n]` has a majority element if more than half its entries are the same. Given an array, design an $O(n \log n)$ algorithm to determine if it has a majority element, and if so, find it. The elements are not sortable (e.g., GIF files), so you can only test for equality `A[i] == A[j]` in $O(1)$ time. Expected output is the majority element or `NO_MAJORITY` if none exists.

**2. How: Algorithm Description**
1. Recursively divide the input array into two equal-sized subarrays, $A_1$ and $A_2$.
2. **Base case:** If the subarray has size 1, report its single element as the majority element.
3. At each level of recursion, check the majority element returned by the left and right subarrays.
4. For each potential majority element, scan the combined array to count its occurrences.
5. If one element occurs strictly more than $n/2$ times in the combined array, return it as the majority element.
6. Otherwise, return `NO_MAJORITY`.

**3. Why: Justification of Correctness**
If a majority element exists in the combined array, it *must* also be the majority element in at least one of its two halves. By recursively dividing the input space, we ensure that if a true majority element exists, it will emerge as a candidate from the base cases up through the recursion. Counting occurrences at each level guarantees we only promote elements that are genuine majorities for the specific subarray being considered.

**4. How Fast: Runtime Analysis**
- We split the problem into 2 subproblems: $a = 2$.
- Each subproblem is exactly half the size: $b = 2$.
- The work to combine results (scanning the array to count occurrences) takes linear time: $O(n^1)$, so $d = 1$.
- We have the recurrence $T(n) = 2T(n/2) + O(n)$.
- Since $\log_b(a) = \log_2(2) = 1$, and $d = 1$, we are in **Case 2 (Balanced)**.
- The overall runtime is $O(n^1 \log n) \implies O(n \log n)$.

### What to remember

- **Problem Structure**: Always define the algorithm steps, provide an informal justification of correctness based on why the problem space shrinks and the base cases hold, and finish with a single Big-O limit.
- **Master Theorem Formula**: $T(n) = aT(n/b) + O(n^d)$
- **Master Theorem Cases**:
  - $d > \log_b(a)$: Root-heavy $\to O(n^d)$ (combine step dominates).
  - $d = \log_b(a)$: Balanced $\to O(n^d \log n)$ (work is spread evenly).
  - $d < \log_b(a)$: Leaf-heavy $\to O(n^{\log_b a})$ (recursive calls dominate).

If Divide & Conquer teaches how recursive calls can split into independent branches, Backtracking is the next useful contrast: recursive branches can also represent competing **choices**.

## Backtracking

In *Algorithms* by Jeff Erickson, he introduces Backtracking as a recursive strategy that constructs solutions incrementally. When the algorithm faces multiple alternatives for the next decision, it recursively evaluates all of them and then chooses the best one.

### Building Solutions Incrementally

My Graduate Algorithms course did not cover Backtracking directly, but I include it here because it highlights an entirely different way to think about recursion. It also helps motivate and frame the following section, Dynamic Programming.

Divide & Conquer is about decomposition and combination: split the problem, solve the independent parts, and combine them. Backtracking is about **search and decision making**. You keep a **partial solution** in your hands, ask what the next logical decision should be, and recursively explore all valid options. 

Erickson's examples make the pattern feel broader than just "`n`-queens and friends":

- In **`n`-queens**, the output is a sequence of queen positions, one per row.
- In a **game tree**, the output is a sequence of moves, one choice per game state.
- In **subset sum**, the output can be viewed as a sequence of include/exclude decisions over the input array.

So this section is less a core syllabus topic than a necessary mental bridge. It explains why some algorithmic ideas naturally start as a raw recursive brute-force search over decisions before they are optimized into something faster.

### The core mental model

Erickson opens with `n`-queens, which makes the mental model concrete. Place queens one row at a time. On row $r$, try each column from left to right. If a square is attacked by an earlier queen, ignore it. If it is legal, place the queen there and recursively try to finish the remaining rows.

That example makes two things clear:

- Every recursive call corresponds to a **legal partial solution**.
- The overall algorithm is just a **depth-first search of the recursion tree** of those partial solutions.

But Erickson adds another point that is even more useful for design: a recursive call needs not only the part of the input you have not processed yet, but also a **summary of past decisions**. The art is figuring out how small that summary can be.

- For **`n`-queens**, you unfortunately need the previously placed queens in full detail.
- For a **game tree**, you often only need the current game state; the exact move history may not matter.
- For **subset sum**, you do not need the exact chosen subset, only the remaining items and the remaining target value.

This is the part that felt missing in my original write-up. Backtracking is not just "try stuff recursively." It is a disciplined way to search an implicit tree where each node stores exactly the information needed to make the **next** decision. Pruning is simply the act of refusing to recurse into obviously illegal branches.

That also explains why exponential runtime is the default. Even if the legality checks prune a lot of branches, the tree can still be enormous.

It also helps explain when Backtracking does **not** naturally become DP. In `n`-queens, the full placement of earlier queens is usually too specific, so two branches rarely land in the exact same state and memoization buys very little. In problems like subset sum, however, many different decision paths can lead to the same reduced state, such as "same remaining items, same remaining target", and that overlap is exactly what DP exploits.

### Discovering repeated work

The most powerful part of this Backtracking mindset is the **reformulation** step. Once you ask what minimal summary of past decisions is required, you often realize the recursive search should solve a slightly more general problem than the one you were originally asked. This generalized problem defines your recursive "state".

In the text segmentation example, you start with a clean recursive idea: choose the first word, and recursively segment the rest of the string. If any split works, return `true`.

At first that sounds like recursion on explicit substrings. But Erickson then rewrites the problem in index form: instead of passing around a suffix `A[i..n]`, treat the original string as global and describe each subproblem only by the starting index `i`. That looks like a minor notation change, but it is a critical habit.

The resulting state is:

$$
\text{Splittable}(i) = \text{whether the suffix } A[i..n] \text{ can be segmented into words}
$$

with base case $\text{Splittable}(n+1)=\text{true}$ and recurrence

$$
\text{Splittable}(i) =
\bigvee_{j=i}^{n}
\Big( \text{IsWord}(i,j) \wedge \text{Splittable}(j+1) \Big)
$$

Once you write it this way, a glaring inefficiency reveals itself: the repeated work stops being hidden. Different branches of the recursion keep rediscovering the exact same suffix questions. "Can I segment starting at index 17?" is the exact same question no matter *how* I reached index 17.

Because the state is so compact (just an index), the algorithm is bound to blindly evaluate the same states over and over. If you chart out the recursive calls, they don't form a continually expanding tree; the branches frequently merge back together into a graph of overlapping states.

That's a helpful insight. First, you identify the next-decision structure. Second, you figure out the smallest summary of past decisions. Finally, you realize this minimal state causes recursive paths to overlap, meaning you can safely save (memoize) the answers instead of recomputing them. That evolutionary leap—from a redundant backtracking search to a cached web of states—is the exact definition of Dynamic Programming.

### What to remember

- **Backtracking works by making one decision per recursive call.** The target object is often naturally viewed as a sequence of decisions.
- **`n`-queens is the canonical picture.** Each node is a legal partial board, and the algorithm is DFS over that implicit tree.
- **The key design question is what summary of past decisions you actually need.** That summary is often the recursive state.
- **Index formulations matter.** Rewriting recursion in terms of compact parameters (`i`, `j`, remaining target) exposes when the search tree revisits the exact same scenario.
- **The backtracking algorithm is often just an intermediate draft.** Its main value is exposing overlapping subproblems before you optimize them.

## Dynamic Programming

After thinking about Backtracking this way, Dynamic Programming feels much less like a separate trick and much more like the next refinement step. You start with a recursive search over choices (Backtracking), identify the smallest state that actually matters, and then notice that many different branches keep asking for the same state again and again.

That is the key difference from Divide & Conquer. In D&C, the recursive calls are meant to be **independent**. In DP, the subproblems **overlap**. The recursive search is still there in spirit, but instead of exploring a huge tree of repeated work, we collapse equivalent calls into a Directed Acyclic Graph (DAG) of states and solve each state once.

So the advantage of DP is not that it avoids recursion conceptually, it is that it avoids recomputation. Once the subproblem definition is right, the rest is bookkeeping: memoize the state, or better yet in this course's style, compute the states bottom-up in a valid topological order.

### Explaining Your Approach (The Output)

There is a distinct difference between *discovering* a DP solution and *communicating* it. Just as we saw in the Divide & Conquer section, skipping straight to pseudocode for your final answer often masks underlying logical flaws. We use the same four-part "What, How, Why, How fast" template introduced earlier to document the final algorithm, but the expectations for each section shift slightly to better fit state definitions and recurrence relations.

**1. What: Problem Specification.** Define the problem you're trying to solve recursively in coherent and precise English. Translate the colloquial statement into sets, sequences, or mathematical terms. State clearly what subset of the input it operates on and any constraints. 

**2. How: Algorithm Description.**
- **(a) Subproblem Definition.** Define the state, like $T(i)$ or $T(i,j)$, purely in words. State clearly what subset of the input it operates on and whether it mandates including a specific element (e.g., "ending at index $i$"). Without a rigorous specification here, it is impossible to determine if your recurrence is correct.
- **(b) Recurrence.** Write the transition purely in mathematical notation. Give a clear recursive formula combining the answers to smaller instances of the exact same problem. A complete recurrence must cover:
  - All recursive step cases.
  - All necessary base cases.
  - Explicit bounds for all variables.

**3. Why: Justification of Correctness.** Explain the logical basis for the recurrence transitions. This acts as an informal inductive proof that your subproblem choices and state transitions correctly build the optimal solution.

**4. How Fast: Implementation Analysis.** To prove the algorithm is efficient and complete, outline:
1. **State Space:** The total number of distinct subproblems in Big-O.
2. **Transition Time:** The runtime to calculate one subproblem, which dictates the total time for the problem as a whole.
3. **Extraction:** Where and how to extract the final answer from the table.
4. **Extraction Time:** The runtime of the extraction step.

> [!note]
> **On Extraction:** Just finding the optimal numeric value is often not enough. If a problem asks for the optimal sequence or path, you must trace back through the table/DAG to reconstruct your choices. This typically takes $O(n)$ or time proportional to the path length.

> [!tip]
> **Implementation Rules of Thumb:** Try to formulate DP as iterative, bottom-up table fills rather than top-down recursion with memoization. Think of the inputs as immutable, and map your base cases starting from index $0$ or $-1$ if necessary to avoid out-of-bounds edge cases.

### Problem-Solving Heuristics

Before you can write out the rigorous 4-part explanation above, you have to actually figure out the algorithm on your scratch pad. Following Jeff Erickson's advice, discovering a DP algorithm is best done in two distinct stages: figuring out the recursion (the *hard* part) and building the table (the *mechanical* part). Ask yourself these questions in order:

**Stage 1: Formulate the Problem Recursively (The Hard Part)**
1. **What is the topology?** Is the input a 1D array, a pair of strings, items with weights, an interval, a tree, or a set of elements?
2. **What are you optimizing?** Max/min value? Count of ways? Length of a subsequence?
3. **What constraint distinguishes the flavor?** Must it be contiguous (Kadane) vs. no-adjacent (Robber) vs. any subsequence (LIS)? Match-based (LCS) vs. cost-based (Edit Dist)? Items used once (0/1) vs. unlimited (Unbounded)?
4. **Write the subproblem in words first.** This is the most important step. A correct recursive formula perfectly follows a clear English specification of the subproblem.
5. **Always define base cases.** Ensure every referenced table entry is well-defined. If $T(i)$ references $T(i-2)$, define $T(0)$ and possibly $T(-1)$.

**Stage 2: Build Solutions Bottom-Up (The Mechanical Part)**
6. **Identify the state space (the memoization structure).** What are all the different ways your recursive algorithm can call itself? This usually comes directly from the smallest summary of past decisions you discovered in the backtracking formulation. The number of combinations dictates the dimensions of your DP table.
7. **Identify dependencies.** Except for base cases, what dictates a cell's value? (e.g., $T(i, j)$ might depend on $T(i-1, j-1)$). This is the point where the backtracking search tree collapses into a Directed Acyclic Graph (DAG) of dependencies.
8. **Find a good evaluation order.** You must compute subproblems *before* the things that depend on them. The dependencies you just identified dictate whether you loop left-to-right, right-to-left, bottom-up, or diagonally. Provide bounds for all variables.
9. **Analyze time.** The total running time is simply the number of distinct subproblems (table size) multiplied by the time to compute a single subproblem.

### Pattern A: Linear (1D Chain)

Linear patterns are the simplest form of Dynamic Programming. The problem space represents a 1D sequence (like an array or time series), and the DAG topology forms a straight chain moving left-to-right. The core question is always: *How does my current element connect to the best solution I've built so far?* The flavor changes depending on how far back you are allowed to look.

The topology is a 1D array. The flavor depends on how far back you look and what you accumulate.

#### Flavor 1: "Look-Back-One" (Contiguous / Kadane's Style)

**1. What: Problem Specification**
Given an array `A[1..n]`, find the maximum sum of any contiguous subarray.

**2. How: Algorithm Description**
**Concept:** _You must extend the immediately preceding subproblem, or start completely over. The current element must be included._

**(a) Subproblem:** T(i) = maximum sum of a contiguous subarray of A[1..i] which includes A[i].

**(b) Recurrence:**

**Base Cases:**

$$
T(1) = A[1]
$$

**Recursive Step:**

$$
\begin{aligned}
T(i) &= \max( A[i],  T(i-1) + A[i] ) \\
     &\quad \text{for } 2 \le i \le n
\end{aligned}
$$

> [!note]
> _Interpretation: either start a new subarray at A[i], or extend the best subarray ending at A[i-1]._

**3. Why: Justification of Correctness**
You must extend the immediately preceding subproblem, or start completely over. By checking both starting a new subarray at A[i] or extending the best subarray ending at A[i-1], we guarantee that the maximum contiguous subarray ending at any index is found. The current element must be included in T(i).

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n)`
- **(2) Fill time:** `O(n)`
- **(3) Return:** `max{ T(i) : 1 ≤ i ≤ n }`
- **(4) Return time:** `O(n)`

---

#### Flavor 2: "Look-Back-Two" (Skip / House Robber Style)

**1. What: Problem Specification**
Given an array of values `A[1..n]`, find the maximum sum obtainable such that no two adjacent elements are selected.

**2. How: Algorithm Description**
**Concept:** _Adjacent elements cannot both be selected. Decide whether to take the current element (skip previous) or leave it (keep previous optimal). The current element need not be included._

**(a) Subproblem:** T(i) = maximum sum obtainable from A[1..i] (A[i] may or may not be selected).

**(b) Recurrence:**

**Base Cases:**

$$
\begin{align*}
T(0) &= 0 \\
T(1) &= A[1]
\end{align*}
$$

**Recursive Step:**

$$
\begin{aligned}
T(i) &= \max( T(i-1),  T(i-2) + A[i] ) \\
     &\quad \text{for } 2 \le i \le n
\end{aligned}
$$

> [!note]
> _T(i-1) = skip A[i]; T(i-2) + A[i] = take A[i] and skip A[i-1]._

**3. Why: Justification of Correctness**
Adjacent elements cannot both be selected. For each step $i$, we decide whether to take the current element (which forces us to skip the previous, taking $T(i-2) + A[i]$) or leave it (keeping the previous optimal $T(i-1)$). The current element need not be included. This exhaustive check ensures we build the optimal non-adjacent sum.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n)`
- **(2) Fill time:** `O(n)`
- **(3) Return:** `T(n)`
- **(4) Return time:** `O(1)`

---

#### Flavor 3: "Look-Back-All" (Subsequence / LIS Style)

**1. What: Problem Specification**
Given an array `A[1..n]`, find the length of the longest increasing subsequence. The sequence need not be contiguous.

**2. How: Algorithm Description**
**Concept:** _The sequence need not be contiguous. Scan all previous valid subproblems to find the best predecessor. The current element must be included._

**(a) Subproblem:** T(i) = length of the longest increasing subsequence of A[1..i] which includes A[i].

**(b) Recurrence:**

**Base Cases:**

$$
T(1) = 1
$$

**Recursive Step:**

$$
\begin{aligned}
T(i) &= 1 + \max \big( \{0\} \cup \{ T(j) : \\
     &\quad 1 \le j < i,\; A[j] < A[i] \} \big) \\
     &\quad \text{for } 2 \le i \le n
\end{aligned}
$$

> [!note]
> _If no j satisfies A[j] < A[i], the max is taken over {0} (set containing zero), so T(i) = 1._

**3. Why: Justification of Correctness**
To find the longest increasing subsequence ending at $i$, we can scan all previous valid subproblems $j$ where $A[j] < A[i]$ to find the best predecessor. If no $j$ satisfies this condition, the max is taken over $\{0\}$, effectively starting a new subsequence of length 1. The sequence need not be contiguous, and the current element must be included to allow extending the sequence.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n)`
- **(2) Fill time:** `O(n²)`
- **(3) Return:** `max{ T(i) : 1 ≤ i ≤ n }`
- **(4) Return time:** `O(n)`

---

### Pattern B: Grid (2D / Strings)

Grid patterns lift the problem space into two dimensions. The constraints usually force movement strictly down, right, or diagonally down-right. This pattern frequently appears when comparing two separate sequences (like strings) against each other, or when physically navigating a 2D matrix. The core question is: *How many distinct paths or cumulative costs lead to my current cell from its immediate neighbors?*

<div class="viz-shell">
  <DPDagVisualizer client:visible pattern="B" />
</div>

The topology is the (i-1, j), (i, j-1), (i-1, j-1) neighborhood. The flavor depends on whether you match elements or accumulate path costs.

#### Flavor 1: "Match or Drop" (Longest Common Subsequence)

**1. What: Problem Specification**
Given two strings `A[1..n]` and `B[1..m]`, find the length of their longest common subsequence.

**2. How: Algorithm Description**
**Concept:** _If elements match, gain 1 and consume both. If not, drop one element from either sequence and take the best._

**(a) Subproblem:** T(i,j) = length of the LCS of A[1..i] and B[1..j].

**(b) Recurrence:**

**Base Cases:**

$$
\begin{align*}
T(0,j) &= 0 \quad \text{for } 0 \le j \le m \\
T(i,0) &= 0 \quad \text{for } 0 \le i \le n
\end{align*}
$$

**Recursive Step:**

$$
T(i,j) =
\begin{aligned}
&\begin{cases}
T(i-1,j-1) + 1 & \text{if } A[i] = B[j] \\
\max( T(i-1,j),  T(i,j-1) ) & \text{if } A[i] \ne B[j]
\end{cases} \\
&\quad \text{for } 1 \le i \le n, 1 \le j \le m
\end{aligned}
$$

**3. Why: Justification of Correctness**
If the current elements match ($A[i] = B[j]$), they can extend the optimal common subsequence, so we gain 1 and consume both. If they do not match, we must drop one element from either sequence and take the best of the two options. This evaluates all possible alignments for a common subsequence.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n·m)`
- **(2) Fill time:** `O(n·m)`
- **(3) Return:** `T(n,m)`
- **(4) Return time:** `O(1)`

---

#### Flavor 2: "Penalty Cost" (Edit Distance)

**1. What: Problem Specification**
Given two strings `A[1..n]` and `B[1..m]`, find the minimum number of edits (insertions, deletions, substitutions) required to transform A into B.

**2. How: Algorithm Description**
**Concept:** _Transform A into B. Every mismatch incurs a cost; take the minimum of substitution, deletion, or insertion._

**(a) Subproblem:** T(i,j) = minimum edit distance to transform A[1..i] into B[1..j].

**(b) Recurrence:**

**Base Cases:**

$$
\begin{align*}
T(0,j) &= j \quad \text{for } 0 \le j \le m \quad (\text{insert } j \text{ characters}) \\
T(i,0) &= i \quad \text{for } 0 \le i \le n \quad (\text{delete } i \text{ characters})
\end{align*}
$$

**Recursive Step:**

$$
T(i,j) =
\begin{cases}
T(i-1,j-1) & \text{if } A[i] = B[j] \\
1 + \min\big( T(i-1,j-1), \\
\quad T(i-1,j), T(i,j-1) \big) & \text{if } A[i] \ne B[j]
\end{cases}
$$

**3. Why: Justification of Correctness**
To transform A into B, we process from the ends. If the current characters match, no cost is incurred, and we reduce both strings. If they mismatch, we must pay a penalty of 1 and choose the cheapest operation: substitution ($i-1, j-1$), deletion ($i-1, j$), or insertion ($i, j-1$). This systematically minimizes total edits over all alignments.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n·m)`
- **(2) Fill time:** `O(n·m)`
- **(3) Return:** `T(n,m)`
- **(4) Return time:** `O(1)`

---

#### Flavor 3: "Path Accumulation" (Min Path Sum)

**1. What: Problem Specification**
Given a grid of costs `C[1..n][1..m]`, you can only move right or down. Find the minimum cost to travel from the top-left cell to the bottom-right cell.

**2. How: Algorithm Description**
**Concept:** _Move through a physical grid accumulating weights. No matching — just movement constraints (right or down)._

**(a) Subproblem:** T(i,j) = minimum cost to reach cell (i,j) from cell (1,1).

**(b) Recurrence:**

**Base Cases:**

$$
\begin{align*}
T(1,1) &= C[1][1] \\
T(i,1) &= T(i-1,1) + C[i][1] \quad \text{for } 2 \le i \le n \\
T(1,j) &= T(1,j-1) + C[1][j] \quad \text{for } 2 \le j \le m
\end{align*}
$$

**Recursive Step:**

$$
\begin{aligned}
T(i,j) &= C[i][j] + \min( T(i-1,j), T(i,j-1) ) \\
       &\quad \text{for } 2 \le i \le n, 2 \le j \le m
\end{aligned}
$$

**3. Why: Justification of Correctness**
Because movement is restricted to right and down, any path reaching cell $(i, j)$ must come from either $(i-1, j)$ or $(i, j-1)$. By iteratively adding the current cell's cost $C[i][j]$ to the cheaper of these two immediate predecessors, we correctly accumulate the minimum path sum to everywhere in the grid.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n·m)`
- **(2) Fill time:** `O(n·m)`
- **(3) Return:** `T(n,m)`
- **(4) Return time:** `O(1)`

---

### Pattern C: Knapsack

Knapsack patterns deal with constrained optimization: you have a limited capacity (weight, budget, etc.) and a menu of items to choose from. The state space is intrinsically 2D—one dimension tracks the items considered so far, while the other tracks the remaining capacity. The defining characteristic of the DAG topology is jumping "leftwards" across the capacity dimension when you choose an item. The core differences lie in item uniqueness (is an item reusable?).

<div class="viz-shell">
  <DPDagVisualizer client:visible pattern="C" />
</div>

The topology involves jumping leftwards across a capacity dimension. The flavor changes based on whether items are unique, reusable, or you are counting combinations.

#### Flavor 1: "Take It or Leave It" (0/1 Knapsack)

**1. What: Problem Specification**
Given `n` items with weights $w_i$ and values $v_i$, and a total weight capacity $W$, find the maximum value of a subset of items that fits in the capacity. Each item can be used at most once.

**2. How: Algorithm Description**
**Concept:** _Each item can only be used once. You transition from the previous row (i-1)._

**(a) Subproblem:** T(i,w) = maximum value achievable using items from {1..i} with capacity w.

**(b) Recurrence:**

**Base Cases:**

$$
T(0,w) = 0 \quad \text{for } 0 \le w \le W
$$

**Recursive Step:**

$$
\begin{aligned}
T(i,w) &= \begin{cases}
T(i-1,w) & \text{if } w_i > w \\
\max( T(i-1,w),  T(i-1, w-w_i) + v_i ) & \text{if } w_i \le w
\end{cases} \\
&\quad \text{for } 1 \le i \le n, 0 \le w \le W
\end{aligned}
$$

> [!note]
> _wᵢ and vᵢ are the weight and value of item i._

**3. Why: Justification of Correctness**
Each item forces a binary choice: either include it or exclude it. If the item's weight exceeds the available capacity $w$, it must be excluded. If it fits, we take the maximum between excluding it ($T(i-1, w)$) or including it ($T(i-1, w-w_i) + v_i$). Both choices transition from the previous row ($i-1$), ensuring each item is used at most once.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n·W)`
- **(2) Fill time:** `O(n·W)`
- **(3) Return:** `T(n,W)`
- **(4) Return time:** `O(1)`

---

#### Flavor 2: "Infinite Reuse" (Unbounded Knapsack / Min Coin Change)

**1. What: Problem Specification**
Given a target amount $W$ and $n$ coin denominations, find the minimum number of coins needed to make change. Each denomination can be used infinitely many times.

**2. How: Algorithm Description**
**Concept:** _If you take an item, you can take it again. You transition from the current row (i), not (i-1). Shown here as Minimum Coin Change._

**(a) Subproblem:** T(j) = minimum number of coins needed to make change for amount j, using coins from the given denominations.

**(b) Recurrence:**

**Base Cases:**

$$
\begin{align*}
T(0) &= 0 \\
T(j) &= \infty \quad \text{if no coin } \le j
\end{align*}
$$

**Recursive Step:**

$$
\begin{aligned}
T(j) &= 1 + \min_{ \substack{1 \le i \le n\\c_i \le j} } \{ T(j - c_i) \} \\
     &\quad \text{for } 1 \le j \le W
\end{aligned}
$$

> [!note]
> _Can also be written as 2D: T(i,w) = min( T(i-1,w), T(i, w-cᵢ) + 1 ), where row i means "using coins 1..i" and staying in row i allows reuse._

**3. Why: Justification of Correctness**
Since coins can be reused infinitely, any valid change configuration for amount $j$ ends with *some* coin $c_i$. To minimize coins used, we take 1 plus the minimum cost of the leftover amount ($j - c_i$). By checking every valid coin $c_i \le j$, we are guaranteed to find the minimum. This can also be written in 2D where staying in row $i$ achieves infinite reuse.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(W)`
- **(2) Fill time:** `O(n·W)`
- **(3) Return:** `T(W)`
- **(4) Return time:** `O(1)`

---

#### Flavor 3: "Counting Combinations" (Subset Sum / Coin Change 2)

**1. What: Problem Specification**
Given a target amount $W$ and $n$ coin denominations, find the total number of ways to make change.

**2. How: Algorithm Description**
**Concept:** _You are not maximizing value — you are summing all possible ways to reach a state. Replace max with +._

**(a) Subproblem:** T(i,w) = number of ways to form amount w using coins from {1..i}.

**(b) Recurrence:**

**Base Cases:**

$$
\begin{align*}
T(0,0) &= 1 \\
T(0,w) &= 0 \quad \text{for } 1 \le w \le W
\end{align*}
$$

**Recursive Step:**

$$
\begin{aligned}
T(i,w) &= \begin{cases}
T(i-1,w) & \text{if } c_i > w \\
T(i-1,w) + T(i, w-c_i) & \text{if } c_i \le w
\end{cases} \\
&\quad \text{for } 1 \le i \le n, 0 \le w \le W
\end{aligned}
$$

> [!note]
> _Staying in row i (for the cᵢ ≤ w case) allows reuse of coin i. Moving to row i-1 only would give 0/1 counting._

**3. Why: Justification of Correctness**
Since we are counting combinations rather than maximizing value, we replace `max` with `+`. The total number of ways to form amount $w$ using coins $1..i$ is exactly the sum of the ways to form $w$ without using coin $c_i$ (moving to $i-1$) plus the ways to form $w - c_i$ using coins $1..i$ (staying in row $i$ to allow reuse).

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n·W)`
- **(2) Fill time:** `O(n·W)`
- **(3) Return:** `T(n,W)`
- **(4) Return time:** `O(1)`

---

### Pattern D: Interval (Pyramid)

Interval patterns deal with optimally merging, splitting, or shrinking a continuous sequence. Unlike linear arrays where you build forward one element at a time, Interval DP considers an entire span `[i..j]` and discovers the optimal way to break it down. The DAG topology works hierarchically—the cost of a large interval depends on the previously computed costs of the smaller sub-intervals contained within it. The differences depend on *where* the split occurs.

<div class="viz-shell">
  <DPDagVisualizer client:visible pattern="D" />
</div>

The topology always merges sub-intervals. The flavor depends on whether you split at an arbitrary k, shrink from edges, or pivot on the last action.

#### Flavor 1: "The K-Split" (Matrix Chain Multiplication / Optimal BST)

**1. What: Problem Specification**
Given a sequence of $n$ matrices, find the most efficient way (minimum scalar multiplications) to multiply these matrices together. Matrix $A_i$ has dimensions $p[i-1] \times p[i]$.

**2. How: Algorithm Description**
**Concept:** _The cost of interval (i,j) depends on splitting it into (i,k) and (k+1,j) plus the cost to merge._

**(a) Subproblem:** T(i,j) = minimum cost to multiply matrices Aᵢ through Aⱼ.

**(b) Recurrence:**

**Base Cases:**

$$
T(i,i) = 0 \quad \text{for } 1 \le i \le n
$$

**Recursive Step:**

$$
\begin{aligned}
T(i,j) &= \min_{ i \le k \le j-1} \{ T(i,k) + T(k+1,j) + p[i-1] \cdot p[k] \cdot p[j] \} \\
       &\quad \text{for } 1 \le i < j \le n
\end{aligned}
$$

> [!note]
> _Fill order: by increasing interval length ℓ = j - i, from ℓ=1 up to ℓ=n-1._

**3. Why: Justification of Correctness**
Any parenthesization of the product $A_i \dots A_j$ must split the product at some matrix $A_k$. The minimum cost is the cost to compute the left half ($T(i,k)$), plus the right half ($T(k+1,j)$), plus the cost to multiply those two resulting matrices ($p[i-1]p[k]p[j]$). By iterating over all possible split points $k$, we find the optimal cost for the interval.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n²)`
- **(2) Fill time:** `O(n³)` (due to testing $O(n)$ splits $k$ per subproblem)
- **(3) Return:** `T(1,n)`
- **(4) Return time:** `O(1)`

---

#### Flavor 2: "Shrink from Edges" (Longest Palindromic Subsequence)

**1. What: Problem Specification**
Given a string $A[1..n]$, find the length of its longest palindromic subsequence.

**2. How: Algorithm Description**
**Concept:** _No k-loop needed. Compare the two outermost elements. If they match, the interval shrinks by 2. If not, shrink by 1 from either side._

**(a) Subproblem:** T(i,j) = length of the longest palindromic subsequence of A[i..j].

**(b) Recurrence:**

**Base Cases:**

$$
T(i,i) = 1 \quad \text{for } 1 \le i \le n
$$

**Recursive Step:**

$$
\begin{aligned}
T(i,j) &= \begin{cases}
T(i+1,j-1) + 2 & \text{if } A[i] = A[j] \\
\max( T(i+1,j),  T(i,j-1) ) & \text{if } A[i] \ne A[j]
\end{cases} \\
&\quad \text{for } 1 \le i < j \le n
\end{aligned}
$$

> [!note]
> _Base case T(i,i+1): if A[i]=A[i+1] then 2, else 1. This is handled by the recurrence if you define T(i,i-1)=0 for empty intervals._

**3. Why: Justification of Correctness**
We don't need to try every arbitrary split point $k$. Instead, we compare the outermost elements. If they match, they can act as the matching ends of a palindrome, contributing $2$ to the length and leaving the inner interval $T(i+1, j-1)$. If they do not match, we shrink the interval from either the left or the right and take the maximum of those two options.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n²)`
- **(2) Fill time:** `O(n²)`
- **(3) Return:** `T(1,n)`
- **(4) Return time:** `O(1)`

---

#### Flavor 3: "The Last Action" (Burst Balloons)

**1. What: Problem Specification**
Given $n$ balloons with values $A[1..n]$, popping balloon $k$ yields $A[k-1] \cdot A[k] \cdot A[k+1]$ coins. Find the maximum coins obtained by bursting all balloons. Balloons $0$ and $n+1$ are virtual with value $1$.

**2. How: Algorithm Description**
**Concept:** _Instead of the first split, think about k as the last balloon popped in interval (i,j). This makes left and right subproblems independent._

**(a) Subproblem:** T(i,j) = maximum coins from bursting all balloons in the open interval (i,j), where i and j are unpopped boundary balloons.

**(b) Recurrence:**

**Base Cases:**

$$
T(i,j) = 0 \quad \text{if } j = i+1 \quad (\text{no balloons between})
$$

**Recursive Step:**

$$
\begin{aligned}
T(i,j) &= \max_{i < k < j} \{ T(i,k) + T(k,j) + A[i] \cdot A[k] \cdot A[j]\} \\
       &\quad \text{for } 0 \le i < j \le n+1
\end{aligned}
$$

> [!note]
> _Boundaries A[0] and A[n+1] are sentinel values = 1. The key insight: choosing k last means the left/right subproblems don't interfere._

**3. Why: Justification of Correctness**
Working backwards, instead of picking the *first* balloon to pop, we pick the *last* balloon $k$ to pop in the interval. By making $k$ the last to pop, the left interval $(i,k)$ and right interval $(k,j)$ become completely independent, as $k$ remains as an unpopped boundary for both. We maximize over all possible choices of the last popped balloon $k$.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n²)`
- **(2) Fill time:** `O(n³)`
- **(3) Return:** `T(0, n+1)`
- **(4) Return time:** `O(1)`

---

### Pattern E: Tree DP

Tree DP adapts dynamic programming to hierarchical, non-linear data structures. Because a subtree's optimal answer only depends on its immediate children, trees perfectly satisfy the optimal substructure requirement. The algorithm almost always follows a recursive post-order traversal: compute the answers for the leaf nodes first, and merge them upward to the root. Tree DP often requires maintaining multiple distinct states (e.g., "included" vs "excluded") per node.

<div class="viz-shell">
  <DPDagVisualizer client:visible pattern="E" />
</div>

Tree DP requires defining multiple states per node to handle inclusion/exclusion constraints. Fill order is post-order (leaves first, root last).

#### Flavor 1: "Independent Set" (Max Weight Independent Set / House Robber III)

**1. What: Problem Specification**
Given a tree where each node $u$ has a weight $w(u)$, find the maximum weight of an independent set (a set of nodes where no two nodes are connected by an edge).

**2. How: Algorithm Description**
**Concept:** _You cannot select a node and its direct children. Two states per node: exclude or include._

**(a) Subproblem:** T(u,0) = max weight of an independent set in the subtree rooted at u, excluding u. T(u,1) = same, including u.

**(b) Recurrence:**

**Base Cases:**

$$
\begin{align*}
T(u,0) &= 0 \\
T(u,1) &= w(u)
\end{align*}
\quad \text{for leaf node } u
$$

**Recursive Step:**

$$
\begin{align*}
T(u,0) &= \sum \max( T(v,0), T(v,1) ) \\
       &\quad \text{over all children } v \text{ of } u \\
T(u,1) &= w(u) + \sum T(v,0) \\
       &\quad \text{over all children } v \text{ of } u
\end{align*}
$$

> [!note]
> _When u is excluded, each child is free to be included or not. When u is included, all children must be excluded._

**3. Why: Justification of Correctness**
In an independent set, if a node is included, its direct children must be excluded. If a node is excluded, its children can be either included or excluded. By maintaining two states for each node $u$, we correctly evaluate both local constraints and propagate the maximum possible weight from the leaves up to the root.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n)  (2 states per node)`
- **(2) Fill time:** `O(n)`
- **(3) Return:** `max( T(root,0), T(root,1) )`
- **(4) Return time:** `O(1)`

---

#### Flavor 2: "Path Tracking" (Diameter of a Tree)

**1. What: Problem Specification**
Given a tree, find the diameter: the length of the longest path between any two nodes.

**2. How: Algorithm Description**
**Concept:** _The longest path may arch over a node. T(u) tracks the longest straight path down from u; a secondary equation tracks the arch._

**(a) Subproblem:** T(u) = length of the longest path starting at u and going downward into the subtree of u.

**(b) Recurrence:**

**Base Cases:**

$$
T(u) = 1 \quad \text{for leaf node } u
$$

**Recursive Step:**

$$
\begin{aligned}
T(u) &= 1 + \max \{ T(v) : v \text{ is a child of } u \} \\
     &\quad (\text{longest arm in nodes})
\end{aligned}
$$

To extract the exact answer for the diameter:

$$
\begin{aligned}
&\text{Diameter through } u = T(v_1) + T(v_2) + 1 \\
&\quad (\text{where } v_1, v_2 \text{ are children with largest } T)
\end{aligned}
$$

$$
\text{Answer} =
\begin{cases}
\max_u \{ \text{diameter through } u \} - 1 & (\text{if asking for edges}) \\
\max_u \{ \text{diameter through } u \} & (\text{if asking for nodes})
\end{cases}
$$

> [!note]
> _If u has ≤1 child, diameter through u is just T(u). Track the global max during the fill pass._

**3. Why: Justification of Correctness**
The longest path (diameter) of a tree either passes through the root of a subtree or does not. By calculating the longest downward path $T(u)$ for every node, we can form an arch through any node $u$ by summing its top two children's downward paths (plus 1). Tracking the maximum arch across all nodes guarantees finding the global diameter. If a node has $\le 1$ child, its diameter through $u$ is just $T(u)$.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(n)`
- **(2) Fill time:** `O(n)`
- **(3) Return:** `max over all u of (top two T(child) values summed)`
- **(4) Return time:** `O(n)`

---

### Pattern F: Bitmask (Subset DP)

Bitmask DP operates on sets and combinations rather than traditional sequences. When the input size is incredibly small ($N \le 20$), an exponential $O(2^N)$ algorithm is often the intended solution. Bitmasks use the bits of an integer to encode which elements have been "used" or "visited" ($0$ for unused, $1$ for used). This allows subsets to become state variables in our recurrences, drastically cutting down on otherwise factorial $O(N!)$ brute-force explorations.

<div class="viz-shell">
  <DPDagVisualizer client:visible pattern="F" />
</div>

Bitmasks track which elements have been visited/used. Subsets are encoded as integers where bit i indicates element i is in the set. The flavor changes based on whether you care about the ending node or just the groups formed.

#### Flavor 1: "Path Building" (Traveling Salesperson Problem)

**1. What: Problem Specification**
Given $n$ cities and the distances between each pair, find the minimum distance to visit every city exactly once and end at a specific city $i$.

**2. How: Algorithm Description**
**Concept:** _You need to know which cities S you have visited AND which city i you are currently at, so you can extend to the next city j._

**(a) Subproblem:** T(S,i) = minimum cost to visit exactly the cities in set S, starting from city 1 and ending at city i, where i ∈ S.

**(b) Recurrence:**

**Base Cases:**

$$
T(\{1\}, 1) = 0
$$

**Recursive Step:**

$$
\begin{aligned}
T(S, i) &= \min_{j \in S\setminus\{i\}} \{ T(S\setminus\{i\}, j) + d(j,i) \} \\
        &\quad \text{for all } S \subseteq \{1..n\} \text{ with } 1 \in S, i \in S, i \ne 1
\end{aligned}
$$

> [!note]
> _S\{i} means the set S with city i removed. Fill order: by increasing |S|. d(j,i) = distance from j to i._

**3. Why: Justification of Correctness**
To find the optimal path covering a subset of cities $S$ ending at $i$, we must have come from some other city $j \in S$. The optimal cost is the minimum over all valid prior cities $j$ of the cost to visit $S \setminus \{i\}$ ending at $j$, plus the distance from $j$ to $i$. This bitmask state tracks exactly which cities are visited to ensure no city is visited twice. Fill order is by increasing $|S|$.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(2ⁿ · n)`
- **(2) Fill time:** `O(2ⁿ · n²)`
- **(3) Return:** `min{ T({1..n}, i) + d(i,1) : 2 ≤ i ≤ n }`
- **(4) Return time:** `O(n)`

#### Flavor 2: "Subset Partitioning" (Minimum Cost to Partition into Groups)

**1. What: Problem Specification**
Given a set of $n$ elements and a cost function for forming a group out of any subset, find the optimal partitioning of all elements into groups to minimize the total cost.

**2. How: Algorithm Description**
**Concept:** _You don't care about order or endpoints — just split the universe into optimal subsets. Iterate over all submasks S' of S._

**(a) Subproblem:** T(S) = minimum cost to optimally partition the elements in set S into groups.

**(b) Recurrence:**

**Base Cases:**

$$
T(\emptyset) = 0
$$

**Recursive Step:**

$$
\begin{aligned}
T(S) &= \min_{\substack{S' \subseteq S \\ S' \ne \emptyset}} \{ T(S \setminus S') + \text{cost}(S') \} \\
     &\quad \text{for all non-empty } S \subseteq \{1..n\}
\end{aligned}
$$

> [!note]
> _The 3ⁿ runtime comes from iterating over all submasks of all masks. Each element is in S', in S\S', or not in S — three choices per element._

**3. Why: Justification of Correctness**
Any partition of set $S$ must contain some group $S'$. The optimal cost to partition $S$ is therefore the minimum over all possible non-empty groups $S' \subseteq S$ of the cost of $S'$ plus the optimal partitioning cost of the remaining elements $S \setminus S'$. Iterating over all submasks systematically evaluates every valid partitioning scheme.

**4. How Fast: Runtime Analysis**
- **(1) Subproblems:** `O(2ⁿ)`
- **(2) Fill time:** `O(3ⁿ)` (due to iterating over all submasks of all masks)
- **(3) Return:** `T({1..n})`
- **(4) Return time:** `O(1)`

### What to remember

- **Topologies**: 1D chain, 2D Grid, Knapsack, Interval (Pyramid), Tree, and Bitmask (Subsets).
- **Subproblem Definition is Key**: T(i) / T(i, j) must be written in English mapping exactly to the components of the desired end state.
- **Four required components**: (a) Subproblem, (b) Recurrence (with base cases & bounds), (c) Time/Space complexity analysis, (d) Return statement.
- **No Recursion Allowed**: Formulate iterative topological sorts, utilizing 1-based indexing, table storage, without modifying immutable input sets.
